JSON Simply Typed


Data is messy and disconnected. JSON-ST makes the data easier to program and connect it, creating a better Web

JSON-ST

JSON-ST is a lightweight Linked Data format. It is easy for developers to program and easy for humans to read and write. It is based on the already successful JSON format and provides a way to help JSON data interoperate at Web-scale. JSON-ST is an ideal data format for programming environments, REST Web services, and unstructured databases such as CouchDB and MongoDB.

Examples

{
 "@type": "https://json-ld.org/contexts/person.jsonld",
 "@id": "http://dbpedia.org/resource/John_Lennon"
 "@value": {
   "name": John Lennon",
   "born": "1940-10-09",
   "spouse": {
          "@ref": "http://dbpedia.org/resource/Cynthia_Lennon"
   }
 },
}

{
  "@id": "https://example.com/listItems?",
  "@value": [
    {
      "@ref": "http://dbpedia.org/resource/John_Lennon"
    },
    {
      "@ref": "http://dbpedia.org/resource/Cynthia_Lennon"
    }
  ]
}


How is JSON-ST different?

JSON-ST aims to be the simplest JSON based linked data format. If formats such RDFa, RDF/XML, N3, Turtle, or JSON-LD are too complex for your project then JSON-ST may be the solution. JSON-ST values simplicity over expressivity.

Syntax

No extra processors or software libraries are necessary to use JSON-ST. JSON-ST specifies only four keywords:

  • @id - Used to uniquely identify things that are being described in the document with URIs

  • @type - Used to set the data type of a node or typed value. Types are uniquely identified with an URI. A node can be assigned only one type

  • @val - Used to specify the data that is associated with a particular property in the graph.

  • @ref - URI used to specify the data that is associated with a particular property in the graph. A smart processor may fetch the resource(@ref) and populates @val field

Rules

  • - Arrays may be untyped

  • - Inner objects may be untyped

  • - @value and @ref are mutually exclusive
